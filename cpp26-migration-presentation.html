  • Contracts (P2900) — First-class pre/postconditions enforced at compile and runtime, replacing fragile asserts
  • Hardened standard library — Bounds checks on operator[], iterators; UB → defined trap
  • Erroneous behavior model — Reading uninitialized variables becomes a well-defined error, not silent UB

⚠ Note: C++26 has no official release yet. The features shown here are based on the latest working draft: eel.is/c++draft. The ISO standard is expected to be finalized in 2026, but the core features (contracts, hardened mode, erroneous behavior) are already voted into the draft and implemented by major compilers.

The Fundamental Shift in C++26

From "developer must remember" to "language makes it impossible to forget"

Before C++26: Safety by Convention

  • Developer must remember to initialize every variable
  • Developer must remember to bounds-check every access
  • Developer must remember to validate every parameter
  • Developer must remember to match printf format strings
  • Violations are silent — compiles fine, crashes at runtime
  • Static analysis catches what humans forgot

C++26: Safety by Design

  • Language initializes by default (erroneous behavior model)
  • Containers check bounds automatically (hardened mode)
  • Functions declare preconditions (contracts: pre/post)
  • I/O is type-checked at compile time (std::format)
  • Violations are compile errors or defined traps
  • Static analysis confirms the code is structurally safe

The safest code is code where the unsafe pattern is structurally impossible — not code where a human remembered to add a check.

🔍 Real Scan Diff: Out-of-Bounds Access

Parasoft BD-PB-COOB + BD-PB-VALRANGE — 3 violations → 0

⚠ Parasoft found: "Possible out of bounds access of empty collection readings_ at index 0" · sensor_data.cpp:29

Pre-C++26 — sensor_data.cpp:28-34

SensorReading SensorData::getReading(size_t index) const { return readings_[index]; ◄── BD-PB-COOB } No bounds check. Empty vector = crash. const SensorReading* SensorData::getReadingsInRange( size_t start, size_t count) const { return &readings_[start]; ◄── BD-PB-VALRANGE } Raw ptr, no size info.

C++26 — sensor_data.cpp:27-37

SensorReading SensorData::getReading(size_t index) const { assert(index < size()); ◄── CONTRACT pre() return readings_.at(index); ◄── HARDENED .at() } Double protection. std::span<const SensorReading> SensorData::getReadingsInRange( size_t start, size_t count) const { assert(start < size()); ◄── CONTRACT pre() assert(start + count <= size()); ◄── CONTRACT pre() return std::span(readings_).subspan(start, count); } ◄── SPAN carries size

3 C++26 mechanisms at work: ① Contract pre() rejects bad input before accessing the container. ② Hardened .at() throws std::out_of_range if the contract is bypassed. ③ std::span replaces raw pointer — the caller always knows the valid bounds. The bug is structurally impossible at 3 independent levels.

🔍 Real Scan Diff: Uninitialized Variable

Parasoft BD-PB-NOTINIT — 1 violation → 0

⚠ Parasoft found: "Non-initialized max_pressure is used" · sensor_data.cpp:59

Pre-C++26 — sensor_data.cpp:50-59

double SensorData::getMaxPressure() const { double max_pressure; ◄── BD-PB-NOTINIT Declared but NOT initialized. for (size_t i = 0; i < readings_.size(); ++i) { if (i == 0 || readings_[i].pressure > max_pressure) { max_pressure = readings_[i].pressure; } } return max_pressure; ◄── UB if readings_ is empty! } Returns garbage value.

C++26 — sensor_data.cpp:50-56

double SensorData::getMaxPressure() const { assert(!empty()); ◄── CONTRACT pre() Empty call = immediate trap. auto it = std::ranges::max_element( readings_, {}, &SensorReading::pressure); ◄── ALGORITHM replaces manual loop double r = it->pressure; No temp variable needed. assert(r > 0.0); ◄── CONTRACT post() return r; Result is always valid. }

The bug was eliminated by removing the pattern entirely. Instead of declaring a variable first and assigning it later (leaving a window for uninitialized reads), C++26 idiom uses std::ranges::max_element — no intermediate variable, no initialization gap. The contract pre(!empty()) prevents the empty-collection case that caused the return of garbage. Two fundamental design changes: algorithms over manual loops + contracts over hope.

C++26 placeholder _ is a language feature for intentional discards (RAII lock guards, structured bindings). Parasoft 2025.2 flags these as BD-PB-VOVR — a known false positive that will be resolved in future rule updates. The real fix: unnecessary variables are simply removed, not hidden behind workarounds.

🔍 Real Code Diff: Type Safety & API Protection

C-style variadics and printf replaced with compile-time type-checked alternatives

Pre-C++26 — config_manager.cpp:24-33

void ConfigManager::loadDefaults(int count, ...) { va_list args; ◄── C VARIADIC va_start(args, count); No type safety at all. for (int i = 0; i < count; ++i) { const char* key = va_arg(args, const char*); const char* val = va_arg(args, const char*); ◄── Wrong type = silent UB printf("Loaded: %s = %s\n", key, val); ◄── printf format string } mismatch = crash va_end(args); }

C++26 — config_manager.h + .cpp

// Type-safe span parameter void ConfigManager::loadDefaults( std::span<const ConfigEntry> defaults) { ◄── SPAN: typed, sized, safe for (const auto& entry : defaults) { config_[entry.key] = entry.value; std::cout << std::format( "Loaded: {} = {}", entry.key, entry.value); } ◄── std::format: compile-time } type checking on args // Old variadic API blocked at compile time: void loadDefaults(int, ...) = delete("Use span"); ◄── = delete("reason") Caller gets clear error msg

Three layers of protection: ① std::span replaces va_list — every parameter is type-checked at compile time. ② std::format replaces printf — format string mismatches are compile errors, not runtime crashes. ③ = delete("reason") blocks the old API with a human-readable error message at the call site.

🔍 Real Code Diff: Buffer Safety & Optional Returns

CWE-120 buffer overflow and silent map insertion eliminated by design

Pre-C++26

// data_processor.cpp — raw memcpy void DataProcessor::copyToBuffer( const double* src, size_t count) { ◄── RAW POINTER: no size info memcpy(buffer_, src, count * sizeof(double)); ◄── count could exceed buffer! buffer_used_ = count; Silent overflow → corruption } // config_manager.cpp — operator[] inserts string ConfigManager::getConfig(const string& key) { return config_[key]; ◄── INSERTS empty "" if missing! } Map grows silently. Memory leak.

C++26

// data_processor.cpp — span + contract void DataProcessor::copyToBuffer( std::span<const double> src) { ◄── SPAN: size is part of type assert(src.size() <= MAX_FILTER_SIZE); ◄── CONTRACT: reject oversized std::ranges::copy(src, buffer_.begin()); ◄── Algorithm, not memcpy } // config_manager.cpp — optional return optional<string> ConfigManager::getConfig( const string& key) const { auto it = config_.find(key); ◄── find(), not operator[] if (it != config_.end()) return it->second; return std::nullopt; ◄── OPTIONAL: explicit "not found" } No insertion. No side effects.

Design-level elimination: memcpy(ptr, n) → ranges::copy(span) makes overflow impossible because span knows its own size. map[key] → optional<T> makes silent insertion impossible because the return type forces the caller to handle "not found".

Current Limitation: GCC 15.x

GCC 16 (with full C++26 contract syntax) is not yet released

⚠️ GCC 16 Status (April 2026)

  • GCC 16 is in Stage 4 — regression fixing only, feature-frozen
  • Release candidate expected late April / May 2026
  • Still has 14 P1 regressions to resolve before RC
  • MSYS2 currently ships GCC 15.2

What GCC 15.x Supports vs. What It Doesn't

✅ Available in GCC 15.2

  • std::format, std::span
  • std::optional, std::ranges
  • Value-initialized members (NSDMIs)
  • Designated initializers
  • = delete("reason")
  • Placeholder _ variables

🚧 Requires GCC 16+

  • pre() / post() contract syntax
  • contract_assert()
  • std::inplace_vector
  • Hardened mode (std::hardened)
  • Full erroneous behavior model

Our approach: We use assert() as a stand-in for contracts in the cpp26 project. Once GCC 16 ships, we'll replace them with the official pre() / post() syntax — a mechanical, low-risk update.
